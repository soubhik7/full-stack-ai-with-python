How remember() and recall() move a fact from conversation to ranked answer

Alice mentions a coffee preference once. Weeks later, a different query about her habits still surfaces it — ranked, scored, and typed — without any conversation ID or session state carried by the caller.

Four signals, one ranked list: how recall() actually scores a memory

A single embedding-similarity number is not enough to rank memories — recency and stated importance matter as much as semantic closeness. Every candidate is scored on all four signals before the list is returned.

query — "What are Carol's dietary restrictions?"

vector similarity

× 0.55

cosine match against the query embedding — top ~200 candidates from Qdrant

cross-encoder rerank

× 0.20

ms-marco-MiniLM-L-6-v2 pairwise relevance, precision pass

recency decay

× 0.15

exp(−Δt / half-life) — type-specific decay rate

stated importance

× 0.10

the 0–10 score set at remember() time

Σ weighted fusion
1.  "I am lactose intolerant" fact · importance 9.0 final 0.87 — sim .81 · rerank .93 · recency .77 · importance .90
2.  "I am allergic to peanuts" fact · importance 10.0 final 0.79 — sim .68 · rerank .84 · recency .64 · importance 1.0
3.  "I prefer vegetarian food and avoid meat" preference · importance 8.0 final 0.52 — sim .58 · rerank .41 · recency .70 · importance .80

Similarity alone would rank differently

Peanut allergy has the highest raw similarity to "dietary restrictions" — rerank and importance are what push lactose intolerance above it here.

BM25 runs in parallel, not shown

A keyword pass catches exact-term matches embeddings under-rank; both feed reciprocal rank fusion before this stage.

Weights are configuration, not code

WEIGHT_SIM / WEIGHT_RERANK / WEIGHT_RECENCY / WEIGHT_IMPORTANCE in .env.

Sleep-Phase Consolidation nightly job · 03:00

What happens while the agent "sleeps": a night's memories, replayed and compressed

The direct computational analogue of hippocampal replay — a nightly pass that decides what today's raw, redundant memories become tomorrow's durable knowledge.

1

Collect — last 24 hours, per user

Query for event / context memories created or updated since the last run.

"Had coffee at 9am" "Meeting at 10am" "Discussed Q4 roadmap"
2

Cluster — group by session, time, and semantic proximity

The three memories above collapse into one cluster: "morning_routine".

3

LLM review — one call per cluster

The model reads the cluster and returns a structured verdict, not free text:

{ "promoted_facts": [{ "id": "3", "reason": "strategic discussion" }], "low_value_memory_ids": ["1"], "synthetic_memories": [{ "text": "Morning: productive session, Q4 planning", "importance": 7.0, "source_ids": ["2","3"] }] }
4

Apply decisions

Low-value memories are archived, promoted facts get an importance bump, and the synthesized summary is written as a new, durable memory.

1 archived 1 promoted 1 synthesized
5

Log the run

A ConsolidationRun record and Prometheus metrics — this poster's row 4 outcome is exactly what gets logged, per user, per night.

Off by default in production

AUTO_CONSOLIDATION_ENABLED=false — it costs one LLM call per cluster, per user, per night.

Nothing is deleted blind

"Low value" memories are archived with provenance, not hard-deleted — the synthetic memory still cites its source IDs.
